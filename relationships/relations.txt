Relational Databases(SQL)

Imagine you're building a system for a small bookstore. You start with a spreadsheet to track orders. This works when you have ten orders. But look at what's happening even at three rows. Maria's name and email appear twice. "Dune" and its author and price appear twice. If Maria changes her email, you have to find and update every row she appears in. If you misspell "Frank Herbert" in one row but not another, you now have inconsistent data. If you delete all orders for "Dune," you lose the fact that the book exists at all.

These aren't minor annoyances. In software, they have names: update anomalies (changing data requires touching multiple rows), redundancy (the same information stored over and over), and deletion anomalies (removing a record accidentally destroys unrelated information). A spreadsheet or CSV has no way to prevent any of these problems.

A relational database solves this by splitting your data into separate, purpose-built tables that reference each other. Instead of one giant spreadsheet, you'd have:

    A customers table (each customer appears exactly once)
    A books table (each book appears exactly once)
    An orders table (connecting customers to books, with quantities and dates)

Now Maria's email lives in one place. "Dune" lives in one place. Orders just point to the customer and book they belong to. Change Maria's email once, and every order she's ever placed automatically reflects the update — because the orders don't store her email, they store a reference to her record.

This is the core idea of a relational database: data is organized into tables, and tables are connected through relationships.
The Building Blocks

Table - A collection of related data organized into rows and columns. Think of each table as representing one thing in your system: customers, products, orders, users, posts. In database terminology, a table is also sometimes called a relation or an entity.

Row (also called a record) - A single entry in a table. One customer. One book. One order. Each row contains all the information about that particular instance.

Column (also called a field or attribute) - A single piece of information that every row in the table has. Name, email, price, date. Every column has a specific data type — you can't put text in an integer column, just like you can't put a string where Python expects an int.

Primary Key - A column (or combination of columns) that uniquely identifies each row. No two rows can have the same primary key value, and it can never be empty. Most of the time, this is an auto-incrementing integer called id. Think of it as a Social Security number for each row - every record gets a unique identifier.

Foreign Key - A column in one table that references the primary key of another table. This is how tables connect. In our bookstore example, the orders table would have a customer_id column that points back to the id column in the customers table. That's a foreign key - it says "this order belongs to customer #7."
When Do You Need a Database?

Not every project needs a database. Here's a simple decision framework:

A CSV or JSON file is fine when:

    Your data fits in memory
    Only one person or process reads/writes at a time
    The data structure is simple and flat
    You're doing one-off analysis (like your Module 2 pandas work)

You need a database when:

    Multiple users or processes access the data simultaneously
    Data has relationships (customers have orders, authors write books)
    You need data to persist reliably beyond a single script run
    You need to enforce rules (this column can't be empty, this email must be unique)
    Your data will grow over time and you need efficient querying

For the AI applications you'll build later in this course - a FastAPI backend with user accounts, a RAG system with stored documents - you absolutely need a database. Files won't cut it.
SQL: The Language of Databases

To talk to a relational database, you use SQL (Structured Query Language, usually pronounced "sequel"). SQL has been around since the 1970s, and it's one of the most enduring languages in all of computing. It doesn't matter if you're using MySQL, PostgreSQL, SQLite, or Microsoft SQL Server - they all speak SQL with minor dialect differences.

SQL lets you do four main things, and they map to an acronym you'll see everywhere: CRUD.

    Create - Insert new data (INSERT)
    Read - Retrieve data (SELECT)
    Update - Modify existing data (UPDATE)
    Delete - Remove data (DELETE)
